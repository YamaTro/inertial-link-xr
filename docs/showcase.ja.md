## 一言でいうと

InertialLink XRは、車体に固定したAndroidスマートフォンを、Unity/OpenXR用の認証付き外部モーション基準として利用するOSSです。アプリが明示的に選んだ背景や周辺視野だけに動きを伝え、ゴーグル本来の頭部トラッキング、Camera、主コンテンツはそのまま残します。

Build Weekで完成したのは、構想図ではなく実際に動くスマートフォン–Unity間の統合です。ただし、人を対象とした乗り物酔いへの有効性試験は行っておらず、医療機器でもありません。

## 課題と仮説

乗客の内耳は車両の旋回や加速を感じていても、ゴーグル内の映像は静止していることがあります。実車環境を使った先行研究では、車両運動と同期する周辺視野や背景の動きによって、酔いの指標が低下した結果が報告されています。一方で、酔い、注意の散りやすさ、没入感の最適なバランスには個人差があり、一つの表示方法が全員に最適とは限りません。

InertialLink XRは、その有効性を再実証したと主張するものではありません。研究や製品へ組み込むために不足していた、次の汎用的な実装層を提供します。

1. スマートフォンを車体へ固定し、頭ではなく車体の運動を測る。
2. ローカル通信でデータを認証・検証する。
3. OpenXRによる通常の頭部追跡を維持する。
4. 上限検査済みのデータを、導入先が選んだコンテンツまたは診断機能だけへ公開する。

2022年HFES、2024年IEEE Access、2017年CHI、2026年Applied Ergonomicsの先行研究は[研究根拠と主張の境界（英語）](research-basis.md)に整理しています。

## 実装したもの

| 層 | Build Weekで完成した内容 |
| --- | --- |
| Android | 加速度、ジャイロ、重力、線形加速度、回転ベクトルを取得し、取付姿勢を変換して送る機能。GPS、カメラ、マイク、アカウント、クラウド、広告、テレメトリーは不使用。 |
| ローカル通信 | 一時的な128ビット鍵、HMAC-SHA-256、時刻同期、再送・期限・値域検査を備えたバージョン管理済みUDP仕様`ILXR/1.0`。Kotlin/C#/Nodeで共通テストベクトルを使用。 |
| Unity/OpenXR | 検証済みサンプルを公開し、明示的に割り当てたコンテンツルートだけを動かすパッケージ。CameraやXR Originを含む階層は拒否。 |
| 視覚デモ | 固定した9:16動画の周囲で、認証済みスマートフォン運動に応じて曲面・透視収束型の方向キューを動かす実演。 |
| 診断 | 現実で測った加速度と仮想加速度を比較し、上限付きの補正候補を報告するだけのモニター。 |
| 再現性 | 依存パッケージ不要のNodeツール、合成記録、異常系テスト、導入ガイド、リリース自動化。 |

## 縦動画デモ

![Xiaomi 13Tの認証済みモーションを受信しながら縦動画とCameraを固定したUnity画面](assets/xiaomi13t-unity-demo.png)

このデモは、乗客がショート動画を見ている場面を想定しています。中央の9:16動画は読みやすい位置に固定します。動きを表現するのは周囲だけで、曲面状に奥行きへ収束する線と星状キューにより、主動画を動かさず進行方向を感覚的に示します。

静止環境でのXiaomi 13T–Unity実機試験では、次を確認しました。

- 認証済みパケット216件を受理。
- 拒否・欠落は0件。
- 時刻同期後の最良往復時間は5.448ミリ秒。
- ローカル動画はフレーム112まで再生。
- 方向キュー2,019個が動作。
- Unity Cameraの姿勢は変化なし。

これは、その試験構成でのデータ経路と描画動作を証明する結果です。快適性、酔いの軽減、VRゴーグルでの互換性、走行中の挙動まで証明するものではありません。

## 二つの基準座標を分離する

```text
車体に固定したAndroid                    導入を選択したUnityアプリ
┌──────────────────────┐  ローカルUDP  ┌────────────────────────┐
│ 車体加速度            ├─────────────►│ 認証済み受信            │
│ 角速度                │  ILXR/1.0    │ 期限・再送・値域検査    │
│ 重力・姿勢            │              │ 指定contentRootのみ      │
└──────────────────────┘              └────────────────────────┘

OpenXRゴーグルの頭部追跡 ─────────────► Camera / XR Origin（変更なし）
```

この分離が設計の中心です。ゴーグルのセンサーだけでは、車体の動きと乗客が意図して動かした頭の動きが混ざります。固定スマートフォンは車体基準を提供し、乗客の頭に対する視点は従来どおりOpenXRが担当します。

## 現実–仮想間のズレを測る

`MotionAlignmentMonitor`は、スマートフォンで測った物理加速度と、アプリが表現する仮想加速度を比較できます。記録済みのUnity合成試験では、次の結果になりました。

| 値 | X軸の結果 |
| --- | ---: |
| 現実で測った加速度 | +1.134 m/s² |
| 意図的に応答を弱くした仮想加速度 | +0.816 m/s² |
| 検出したズレ | 0.318 m/s² |
| 上限付き補正候補 | +0.159 m/s² |

サンプル式は`clamp((measured - virtual) × 0.5, -2.0, +2.0)`です。結果は診断表示だけに使われ、車両、モーションプラットフォーム、Camera、XR Origin、アプリ内Transformを自動操作しません。

## セキュリティとプライバシー

モーションパケットを信頼済みとは扱いません。受信側は安全側に失敗します。

- 鍵が違う、またはHMACが改ざんされたパケットを拒否。
- 再送、古い時刻、不自然に未来の時刻を拒否。
- NaN、無限大、値域外の値を拒否。
- 無効な入力では安全タイマーを更新しない。
- 有効な入力が止まると、最後の動きを保持せず滑らかに中立へ戻す。

通常の経路はローカルかつ最小限で、探索ブロードキャスト、GPS、アカウント、クラウド、テレメトリー、広告、カメラ、マイクを使いません。HMACは認証用で暗号化ではないため、動きデータの秘匿性が必要なら専用リンクか信頼できるトンネルが必要です。[脅威モデル（英語）](threat-model.md)と[セキュリティポリシー](../SECURITY.md)も参照してください。

## 乗り物酔い以外の将来性

再利用できる本質は、一つの視覚効果ではなく「認証済みの物理モーション基準」です。将来の応用候補には次があります。

- **デジタルツインの位置合わせ：** 実機の運動と仮想側を比較し、ズレを見つける。
- **シミュレーター較正：** 訓練シーンが測定済みの物理運動をどの程度表現できているか数値化する。
- **モーションプラットフォームの品質確認：** 指令値・描画値と、独立した実測基準を比較する。
- **移動空間型XR：** 頭部追跡を置き換えず、移動する客室の運動に環境効果を同期する。
- **研究用計測基盤：** 周辺キューの形、強度、極性、しきい値、遅延補正を比較する。

これらは技術的に展開できる方向であり、全用途の製品実装や検証が完了したという主張ではありません。

## 安全に試す

最初は[日本語README](../README.ja.md)のループバック合成試験を使い、その後に[Unity導入](integration.ja.md)、[Android準備](android-setup.ja.md)、[較正チェックリスト](calibration.ja.md)の順で進めてください。公道や公共交通機関ではなく、合成入力と静止環境から始めてください。

同乗者専用です。不快感があれば直ちに中止してください。運転中、自転車走行中、歩行中、機械操作中にゴーグルを使用・設定しないでください。
